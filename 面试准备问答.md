# Gamer Beta 项目面试准备问答

> 适用场景：面试官围绕项目架构、核心实现、技术选型、问题排查、个人思考进行追问。回答时建议先讲业务目标，再讲架构分层和关键链路，最后补充限制与后续优化。

## 一、项目整体介绍

### 1. 请你简单介绍一下这个项目。

Gamer Beta 是一个基于 Web 的互动小游戏创作与分享平台。用户登录后可以在 Create 页面输入游戏创意，并上传图片、视频或文本文件作为参考素材，后端通过 Multi-Agent 生成流程把用户需求转成结构化游戏设计，再生成单文件 HTML5 Canvas 小游戏，保存到本地 uploads 目录并写入 SQLite 数据库。前端可以直接通过 iframe 预览生成结果，也可以进入详情页和播放页访问游戏。

项目的重点不是单纯展示游戏列表，而是把“创意输入、素材上传、AI 生成、代码校验、失败兜底、平台发布”串成一个完整闭环。

### 2. 这个项目的技术栈是什么？

前端使用 Vue 3、Vite、Vue Router、Pinia 和 Axios。Vue 负责页面组件，Vue Router 负责单页应用路由，Pinia 管理登录态，Axios 统一封装 API 请求和 JWT 注入。

后端使用 Node.js + Express.js，数据库使用 better-sqlite3 操作 SQLite，文件上传使用 multer，认证使用 JWT 和 bcryptjs，AI 接入使用 OpenAI Node SDK，并兼容自定义 baseURL，例如 OpenAI 官方接口或 DeepSeek 这类兼容接口。

游戏运行层使用单文件 HTML5 Canvas，生成后的游戏以静态 HTML 文件形式挂在 `/uploads/games/*.html` 下访问。

### 3. 项目的核心业务流程是什么？

核心流程是：

1. 用户在 Create 页面输入 prompt，并可上传附件。
2. 前端先调用 `/api/upload/file` 上传附件，拿到 url、filename、mimetype、size、kind 等元数据。
3. 用户点击发送后，前端调用 `/api/games/generate`，请求体包含 prompt 和 attachments。
4. 后端 `gameController.generateGame` 做参数校验，然后调用 `orchestrateGameGeneration`。
5. Orchestrator 先调用 Game Designer Agent 生成结构化游戏设计 JSON。
6. 再调用 Code Generator Agent 生成完整 HTML5 Canvas 游戏。
7. 后端校验生成 HTML 是否包含 canvas、start 控件、完整 script/html，并用 `vm.Script` 做 JavaScript 语法校验。
8. 校验失败则请求模型修复一次；仍失败则使用本地 fallback 游戏模板。
9. 生成的 HTML 写入 uploads/games，游戏基本信息写入 `games` 表，设计和附件元数据写入 `game_meta` 表。
10. 前端收到 gameUrl 后通过 iframe 预览，并跳转到详情/播放页面。

## 二、架构设计

### 4. 你是怎么做前后端分层的？

前端主要负责用户交互、文件选择、生成状态展示、预览和页面路由；后端负责认证、文件上传、AI 调用、生成编排、文件落盘和数据库持久化。前端不直接参与 AI prompt 拼接，也不处理生成代码的校验逻辑，这些都集中在后端服务层，避免把关键业务逻辑散落到浏览器端。

后端内部也做了分层：routes 只定义接口入口，controllers 负责参数校验和响应格式，services 负责 AI 生成和编排，models/db.js 负责数据库初始化和连接。

### 5. 后端为什么要设计 Orchestrator？

因为 AI 生成不是一个单点调用，而是一条流水线：素材归一化、游戏设计、代码生成、文件保存、数据库写入、步骤状态返回都需要统一协调。`orchestrator.js` 把这些步骤集中起来，Controller 就不需要关心具体生成细节。

这样做的好处是职责清晰，也方便扩展。例如后续要增加任务队列、生成历史、重试机制、日志追踪，只需要围绕 Orchestrator 改造，而不用大规模改动接口层。

### 6. Multi-Agent 在这个项目中具体指什么？

这里的 Multi-Agent 是业务职责上的多 Agent，而不是多个进程。主要分成两个 Agent：

1. Game Designer Agent：把用户 prompt 和附件上下文转成结构化游戏设计 JSON，包括 title、genre、mechanics、controls、winCondition、visualStyle、features、assetUsage 等字段。
2. Code Generator Agent：根据结构化设计生成完整、可运行的单文件 HTML5 Canvas 游戏。

中间用 Orchestrator 串联，先设计再生成代码。这样比直接让模型从 prompt 输出完整代码更稳定，因为结构化设计给代码生成阶段提供了更明确的约束。

### 7. 为什么不让模型一步直接生成游戏代码？

一步生成虽然简单，但可控性差。模型可能遗漏玩法、控件、胜负条件，也不方便保存游戏元数据。拆成设计和代码两个阶段后，第一阶段输出 JSON 可以被平台记录和展示，第二阶段只专注实现，生成质量更容易控制。

此外，如果生成失败，也能判断是设计阶段失败还是代码阶段失败，方便日志定位和后续扩展。

### 8. 前端 Create 页面是怎么组织交互的？

Create 页面采用聊天式输入加右侧预览布局。左侧负责 prompt、快捷提示、附件上传、生成步骤展示；右侧使用 iframe 预览生成的游戏，并允许用户查看标题、描述和标签。

附件上传是先上传后生成：`handleFileSelect` 把文件通过 FormData 发给 `/api/upload/file`，成功后存入 `attachments`。发送时 `sendMessage` 会复制当前附件列表，连同 prompt 一起提交给 `/api/games/generate`。

### 9. 生成步骤为什么要返回给前端？

AI 生成耗时较长，如果前端只是一直 loading，用户不知道系统卡在哪里。后端返回 `steps`，包含 Game Designer、Code Generator、File Storage、Database 等阶段的状态和描述，前端可以展示 Multi-Agent Pipeline，让用户理解生成过程，也方便调试。

当前实现是请求完成后一次性返回步骤结果，后续如果要优化体验，可以改成 SSE 或 WebSocket 实时推送阶段状态。

## 三、AI 生成与多模态处理

### 10. openaiClient.js 的作用是什么？

`openaiClient.js` 封装了 OpenAI SDK 的调用，统一读取 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`OPENAI_MODEL` 等环境变量。它提供普通文本 chat、多图片 chat、附件 chat、图片转 base64、文本文件读取等能力。

这个文件还负责判断当前 endpoint 是否支持图片输入。如果 baseURL 是 OpenAI 官方 `/v1`，就允许发送 `image_url`；如果是 DeepSeek 等文本兼容接口，就降级为纯文本上下文，避免发送不兼容的多模态消息导致接口报错。

### 11. 项目如何兼容 DeepSeek 这类纯文本接口？

核心是 `SUPPORTS_IMAGE_INPUT` 判断。只有当 `OPENAI_BASE_URL` 匹配 OpenAI 官方 endpoint 时，后端才会把图片转成 base64 data URI 并作为 `image_url` 发送给模型。

如果是 DeepSeek 这类文本模型，后端不会发送 `image_url`，而是把附件文件名、类型、URL 等信息写进文本 prompt。这样模型不能真正理解图片像素，但仍能知道用户上传了哪些素材，并在生成游戏代码时使用这些 URL 把图片画到 Canvas 上。

### 12. 上传图片在生成游戏里是如何真正被使用的？

图片附件会经过两个阶段：

第一，在 Game Designer 阶段，图片会作为视觉参考，要求模型规划如何把图片用作背景、玩家、敌人、道具或 UI 元素。

第二，在 Code Generator 阶段，`buildAssetPrompt` 会明确列出图片 URL，并要求模型使用 `new Image()` 预加载这些 URL，在 Canvas 中绘制至少一张上传图片。fallback 模板里也会读取 image assets，按文件名猜测 background、hero、enemy、coin 等用途，并在 Canvas 中显示。

### 13. 文本文件附件是怎么处理的？

如果附件是 text、json、csv、md 等文本类文件，Game Designer 会通过 `loadTextFile` 从 uploads 目录读取内容，并截取前 12000 个字符注入 prompt，作为游戏设定、规则或剧情参考。

这样可以让用户上传规则文档、关卡配置、剧情草稿等资料，但又通过长度截断控制 prompt 体积。

### 14. 视频附件目前是怎么处理的？

当前项目没有做视频抽帧、语音转写或逐帧理解。视频会作为附件元数据传入模型，主要根据文件名、类型和用户 prompt 作为动作风格、节奏、视觉参考。技术文档里也明确把这列为当前限制。

如果要进一步增强，可以用 ffmpeg 抽取关键帧，再把关键帧作为图片输入；也可以接入语音识别，把音频转成文本上下文。

### 15. 为什么要让模型输出 ASCII UI 文本？

在生成 HTML/JS 时，如果模型输出 emoji 或非 ASCII 字符，有时会因为编码、截断或转义问题破坏 JavaScript 字符串。`codeGenerator.js` 的系统提示里明确要求使用 plain ASCII text，目的是降低生成代码在保存和执行时出现乱码或语法错误的概率。

这个约束是实际问题驱动的：生成类项目里，不稳定因素很多，减少编码风险能提升成功率。

## 四、生成质量保障与异常处理

### 16. 如何判断生成的游戏是否可运行？

`validateHtmlGame` 做了几类检查：

1. HTML 中必须包含 `<canvas>`。
2. 内容中必须出现 start 相关控件。
3. 必须包含 `</script>` 和 `</html>`，防止模型输出被截断。
4. 抽取所有 script 内容，用 Node 的 `vm.Script` 做 JavaScript 语法检查。

这些检查不能保证游戏逻辑一定好玩，但能拦截最常见的不可启动问题，例如缺少 canvas、Start 按钮不可见、HTML 截断、JS 语法错误。

### 17. 生成失败后怎么处理？

生成失败分两层处理：

第一层是模型修复。如果初次生成的 HTML 没通过校验，后端会调用 `repairGameCode`，把错误信息和原始 HTML 发回模型，要求返回完整修复后的 HTML。

第二层是本地 fallback。如果修复仍失败，就使用 `fallbackGameTemplate` 生成一个本地稳定模板，保证 Start/Restart 可点击、游戏循环可运行，并尽量显示上传的图片素材。

这种设计的目标是保证用户每次生成至少能得到一个可运行结果，而不是直接报错。

### 18. 为什么 fallback 模板很重要？

AI 生成代码具有不确定性，尤其是长 HTML 文件可能出现截断、事件绑定错误、DOM 顺序错误、语法错误。fallback 模板相当于最后一道工程保障：即使模型输出不可用，平台仍能发布一个可玩的基础游戏。

从产品角度，它减少用户挫败感；从面试角度，它体现了我没有完全依赖模型能力，而是用工程手段兜底生成质量。

### 19. `vm.Script` 校验有什么优点和局限？

优点是可以在服务端快速发现 JavaScript 语法错误，不需要启动浏览器。它适合拦截括号不闭合、字符串截断、非法语法等问题。

局限是它只能检查语法，不能保证 DOM 元素存在、事件绑定真实生效、Canvas 绘制正常，也不能模拟用户点击。后续可以加 Playwright 做端到端校验，例如加载生成 HTML、点击 Start、检查 canvas 是否非空。

### 20. 如何防止模型输出 Markdown 代码块影响保存？

`cleanHtml` 会去掉开头或结尾的 ```html、``` 等 Markdown fence，并尝试从响应中定位 `<!DOCTYPE html>`。这是因为模型经常习惯把代码包在 Markdown 代码块里，而生成结果需要直接保存成可访问的 HTML 文件。

### 21. 你遇到过什么比较典型的问题？

比较典型的问题有：

1. 模型输出被截断，导致缺少 `</script>` 或 `</html>`，游戏无法加载。
2. Start 按钮存在但没有正确绑定事件，点击后游戏状态不切换。
3. 使用 DeepSeek 这类纯文本 endpoint 时，如果仍然发送 `image_url` 消息，会导致接口不兼容。
4. 非 ASCII 字符或 emoji 出现在 JS 字符串里时，可能引入编码和解析问题。
5. 上传图片只是被模型“描述”，但没有实际显示到游戏里，因此需要在 prompt 和 fallback 中强制使用图片 URL。

解决思路是：增加 endpoint 能力判断、生成后校验、模型修复、fallback 模板、ASCII 约束，以及在 prompt 中明确要求 `new Image()` 加载上传素材。

## 五、数据库与接口设计

### 22. 数据库表是怎么设计的？

主要有五张表：

1. `users`：用户账号、密码哈希、昵称、头像、GitHub OAuth 信息。
2. `games`：游戏标题、描述、封面、游戏文件 URL、标签、作者、播放次数、状态。
3. `game_meta`：游戏生成相关的结构化元数据，以 JSON 字符串保存。
4. `assets`：素材库文件信息。
5. `favorites`：用户收藏游戏的关系表。

`games` 保存列表和详情页常用字段，`game_meta` 保存扩展信息，避免频繁改表。

### 23. 为什么 `game_meta` 使用 JSON，而不是把所有字段拆成列？

生成元数据字段变化较快，例如 mechanics、controls、features、attachments、referenceImages、assetUsage 等，后续还可能增加 agent 日志、token 消耗、模型参数。如果都拆成列，schema 会频繁变动。

用 `game_meta.meta_json` 可以提高迭代灵活性。缺点是查询某个 meta 字段不如结构化列方便。当前项目更重视生成记录和详情展示，所以 JSON 是比较合适的折中。

### 24. 为什么使用 SQLite？

这个项目是本地演示和面试项目，SQLite 部署简单，不需要额外数据库服务，适合快速开发和交付。`better-sqlite3` 是同步 API，使用简单，配合 WAL 和外键约束可以满足当前规模。

如果进入生产环境，SQLite 的并发写能力和运维能力会成为限制，可以迁移到 PostgreSQL 或 MySQL，并增加迁移工具和连接池。

### 25. JWT 认证是怎么实现的？

用户注册或登录后，后端用 `jsonwebtoken` 签发包含 `userId` 的 token，有效期 7 天。前端 Pinia store 把 token 保存到 localStorage。Axios 请求拦截器每次请求自动添加 `Authorization: Bearer <token>`。

后端 `authMiddleware` 校验 token，成功后把 `req.userId` 注入请求对象，供上传、生成、删除、收藏等接口使用。

### 26. 密码安全是怎么处理的？

注册时后端用 bcryptjs 对密码做 hash，数据库只保存 `password_hash`，登录时通过 `bcrypt.compareSync` 校验，不保存明文密码。同时注册阶段做了邮箱格式校验和最小密码长度校验。

生产环境还应要求强 `JWT_SECRET`、HTTPS、限流、验证码或登录失败次数限制。

### 27. 文件上传是怎么设计的？

上传使用 multer 的 diskStorage。文件名用 UUID 加原扩展名，避免用户文件名冲突。图片或 cover 文件落到 `uploads/covers`，游戏和普通文件落到 `uploads/games`。Express 通过 `/uploads` 静态托管这些文件。

`/api/upload/file` 是通用附件入口，会返回 url、filename、originalName、mimetype、size、kind，前端再把这些元数据传给生成接口。

### 28. 游戏列表接口支持哪些查询？

`GET /api/games` 支持分页、tag 过滤和 search 搜索。后端用 SQL 拼接查询条件，但参数都通过 prepared statement 绑定，避免 SQL 注入。返回时会把数据库中的逗号分隔 tags 转成数组，并附带作者信息。

### 29. 播放次数是怎么统计的？

`getGame` 查询游戏详情后，会执行 `UPDATE games SET play_count = play_count + 1 WHERE id = ?`。响应里的 playCount 返回的是原值加一。这个实现简单直接，适合当前规模。

如果是高并发生产场景，可以考虑异步计数、Redis 聚合、去重统计或防刷策略。

### 30. 收藏功能如何保证一个用户不能重复收藏同一个游戏？

`favorites` 表对 `(user_id, game_id)` 建了唯一约束。接口 `POST /api/favorites/:gameId` 实现 toggle：如果已存在则删除，如果不存在则插入。数据库唯一约束可以从底层保证不会重复收藏。

## 六、前端实现

### 31. Pinia 在项目里主要做什么？

Pinia 主要管理认证状态，包括 `user`、`token`、`isLoggedIn`、login、register、fetchUser、logout 等。这样登录状态可以在多个页面共享，例如 Create 和 Assets 页面需要检查是否登录。

### 32. Axios 封装解决了什么问题？

`client/src/api/index.js` 创建了统一 Axios 实例，baseURL 是 `/api`，并设置默认超时时间。请求拦截器从 localStorage 读取 token 并注入 Authorization 头，响应拦截器遇到 401 会清除 token。

这样业务组件不需要重复写 token 注入逻辑。

### 33. 前端路由如何处理登录限制？

路由表中 Create 和 Assets 设置了 `meta: { requiresAuth: true }`。当前 Create 页面自身在 `onMounted` 里判断 `auth.isLoggedIn`，未登录则跳转 `/login`。

可以改进的地方是增加全局 router guard，把所有需要登录的页面统一拦截，而不是分散在页面组件里。

### 34. 为什么预览生成游戏使用 iframe？

生成的游戏是完整 HTML 文件，使用 iframe 可以把它和 Vue 应用隔离开，避免生成游戏的 CSS/JS 影响主应用。Create 页面还给 iframe 设置了 sandbox，允许脚本和同源访问，降低生成 HTML 对主页面的影响。

后续可以进一步收紧 sandbox 权限，并对生成 HTML 做安全过滤。

## 七、安全性与风险

### 35. 这个项目有哪些安全风险？

主要风险包括：

1. 生成的 HTML/JS 是模型输出，属于不可信代码，虽然放在 iframe 里，但仍要注意 XSS 和同源权限。
2. `/uploads` 静态托管用户上传文件，如果允许 HTML 上传，可能被直接访问执行。
3. 当前默认 `JWT_SECRET || 'secret'` 适合开发，不适合生产。
4. 文件删除时部分路径处理比较简单，需要更严格限制只能删除 uploads 下文件。
5. 文件上传需要更严格的 MIME、扩展名、大小和内容校验。
6. AI 接口调用缺少用户级限流和成本控制。

### 36. 如果上线生产，你会怎么加强安全？

我会做以下改造：

1. 强制配置安全的 `JWT_SECRET`，去掉默认 secret。
2. 使用对象存储保存上传文件，并限制可执行类型。
3. 生成游戏放到独立域名或隔离子域名，避免与主站共享敏感 cookie/localStorage。
4. iframe sandbox 移除不必要权限。
5. 对上传文件做白名单、病毒扫描和内容类型校验。
6. 增加接口限流、生成配额、日志审计和异常告警。
7. 删除文件时做路径归一化校验，确保只能操作 uploads 目录内文件。

### 37. 直接把生成 HTML 存到服务器有什么问题？

优点是简单，访问也方便。但问题是生成 HTML 本质上是可执行内容，如果与主应用同源，安全隔离不够强。生产环境更好的做法是把游戏文件放到独立静态资源域名，或对象存储加 CDN，并通过严格 CSP 和 iframe sandbox 限制能力。

### 38. 本项目里有哪些地方体现了防 SQL 注入？

数据库查询基本使用 `db.prepare(...).get/all/run(...)` 的参数绑定，而不是把用户输入直接拼到 SQL 里。例如搜索、标签、详情查询、收藏操作都通过 `?` 占位符传参。

需要注意的是，动态拼接 SQL 条件本身没问题，关键是用户输入部分必须通过参数绑定传入。

## 八、性能与可扩展性

### 39. 当前架构的性能瓶颈在哪里？

主要瓶颈在 AI 生成调用，因为模型响应慢且不稳定；其次是本地文件系统和 SQLite 在并发写入场景下能力有限。生成 HTML 的校验本身开销不大，但如果并发生成多，Node 单进程和同步 SQLite 写入会限制吞吐。

### 40. 如何把生成流程改造成异步任务？

可以新增 `generation_tasks` 表，用户提交生成请求后立即返回 taskId。后端用队列处理任务，任务状态包括 pending、designing、generating、repairing、saving、done、failed。前端通过轮询、SSE 或 WebSocket 获取状态。

这样可以避免 HTTP 长连接超时，也方便做失败重试、历史记录和生成日志。

### 41. 为什么当前没有使用 Phaser.js？

当前目标是让模型生成单文件 HTML5 Canvas 游戏，依赖少、保存和预览简单。Phaser.js 能提升游戏工程稳定性和能力上限，但引入框架后 prompt 约束、资源加载、版本兼容都会复杂一些。

后续可以引入 Phaser 模板，让模型只填充玩法逻辑或配置，这会比完全自由生成 HTML 更稳定。

### 42. 如果用户并发生成很多游戏，会出现什么问题？

可能出现 AI API 请求堆积、接口超时、SQLite 写锁竞争、本地磁盘占用增长、uploads 文件清理困难、成本不可控等问题。

解决方案包括任务队列、并发限制、用户配额、生成历史、对象存储、后台清理任务、成本统计和失败重试机制。

### 43. 如何做 token/cost 统计？

OpenAI 响应通常会返回 usage 字段，可以在 `openaiClient.js` 里统一记录 prompt_tokens、completion_tokens、total_tokens、model、userId、taskId。数据库可以新增 `ai_usage` 表或写入 generation task meta。

这样可以按用户、模型、日期统计成本，也能对异常高消耗用户做限制。

## 九、项目限制与后续优化

### 44. 当前项目最主要的限制是什么？

第一，多模态能力取决于模型 endpoint。DeepSeek 这类文本接口无法真正理解图片像素，只能使用文件名、类型和 URL。

第二，视频没有抽帧和转写。

第三，fallback 游戏模板玩法比较通用，只保证可运行，不保证完全符合用户创意。

第四，没有生成任务历史和版本管理。

第五，本地 uploads 和 SQLite 适合演示，不适合大规模生产。

### 45. 如果继续迭代，你最想优化哪里？

我会优先优化生成任务系统和游戏质量验证。任务系统可以解决长请求、重试、历史记录和日志问题；质量验证可以通过 Playwright 自动打开生成游戏、点击 Start、检查 canvas 是否绘制内容，从“语法可运行”提升到“基本可交互”。

然后再做对象存储、配额计费、多模态增强和 Phaser 模板化生成。

### 46. 项目里有没有可以重构的地方？

有。比如路由登录保护可以从页面内判断升级为全局 router guard；上传模块和素材库模块可以更好地统一，因为现在 `/api/upload/file` 上传后不会自动写入 assets 表；生成步骤可以持久化成任务表，而不是只在响应里返回；删除文件时需要更严格的路径安全封装。

这些不是当前演示版本的阻塞问题，但都是工程化演进方向。

### 47. 如何解释“这是一个 AI 项目，但不是简单套壳”？

这个项目的 AI 调用只是核心能力之一，工程难点在于把不稳定的模型输出产品化。项目里做了结构化设计阶段、代码生成阶段、附件归一化、多模态兼容、HTML 清洗、JS 语法校验、模型修复、本地 fallback、文件发布和数据库记录。

所以它不是简单把用户 prompt 发给模型，而是围绕 AI 生成建立了一套可落地、可恢复、可展示的工程流水线。

## 十、可主动展示的代码点

### 48. 如果面试官让你指代码讲架构，你可以怎么讲？

可以按这条线讲：

1. `client/src/views/Create.vue`：用户输入、附件上传、生成请求、iframe 预览。
2. `client/src/api/index.js`：Axios baseURL、JWT 自动注入、401 处理。
3. `server/src/routes/games.js`：`POST /api/games/generate` 路由入口。
4. `server/src/controllers/gameController.js`：校验 prompt，调用生成编排。
5. `server/src/services/orchestrator.js`：Multi-Agent 流水线，保存文件和数据库。
6. `server/src/services/gameDesigner.js`：把 prompt 和附件变成结构化设计 JSON。
7. `server/src/services/codeGenerator.js`：生成 HTML、校验、修复、fallback。
8. `server/src/services/openaiClient.js`：OpenAI SDK 封装和多模态兼容。
9. `server/src/models/db.js`：SQLite schema 和 WAL/外键配置。

### 49. 面试官问“你本人在项目里的思考是什么”，可以怎么回答？

我重点思考的是如何把不确定的 AI 输出变成相对稳定的用户体验。模型生成小游戏代码这件事天然不稳定，所以我没有只做一个 prompt 调用，而是拆成设计、生成、校验、修复、兜底几个环节。

另外，我也考虑了模型供应商兼容问题。OpenAI 官方接口支持图片输入，但 DeepSeek 这类接口可能只支持文本，所以我在后端做了 endpoint 能力判断，不能发图片时就退化成文本附件上下文，同时仍然让生成代码引用图片 URL。

最后，我把生成结果存成标准 HTML 文件并写入数据库，保证它能被平台列表、详情、播放、收藏等普通业务功能复用，而不是停留在一次性的 AI demo。

### 50. 如果面试官问“这个项目最难的地方是什么”，可以怎么回答？

最难的是生成结果的稳定性。模型可能输出不完整 HTML、错误 JS、不可点击的 Start 按钮，或者没有真正使用用户上传的图片。我的处理方式是：

1. 在 prompt 层面明确要求单文件 HTML、Canvas、Start/Restart、ASCII 文本、`new Image()` 引用上传图片。
2. 在代码层面对生成结果做 HTML 完整性和 JS 语法校验。
3. 校验失败后让模型带着错误信息修复一次。
4. 修复失败时使用本地 fallback 模板保证可运行。

这个过程体现了工程上对 AI 不确定性的控制。

## 十一、简短版项目介绍

如果面试官只给 1 分钟，可以这样说：

Gamer Beta 是一个 AI 小游戏生成和分享平台。前端用 Vue 3 + Vite，后端用 Express + SQLite。用户在 Create 页面输入创意并上传素材，后端先通过 Game Designer Agent 生成结构化游戏设计，再通过 Code Generator Agent 生成单文件 HTML5 Canvas 游戏。生成结果会经过 HTML 完整性检查和 `vm.Script` 语法校验，失败时会尝试模型修复，仍失败则用本地 fallback 模板保证游戏可运行。生成的 HTML 文件保存到 uploads，游戏信息和生成元数据写入 SQLite，前端通过 iframe 预览，并支持列表、详情、播放、收藏等平台功能。项目的核心价值是把不稳定的 AI 代码生成做成一个有校验、有兜底、可发布的工程闭环。
